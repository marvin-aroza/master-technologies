i18n & l10n

Internationalization and localization strategies in Angular.

@angular/localize (Built-in i18n)

Angular's native approach operates at compile-time (AOT). The compiler generates a completely separate application bundle for each locale. This provides the best performance because there's no runtime translation parsing.

  • i18n attribute: Mark HTML elements for translation: `

    Hello

    `.
  • $localize: Translate strings in TypeScript: `const title = $localize\`:@@homeTitle:Hello\`;`.
  • ng extract-i18n: CLI command that generates an XLIFF/XMB file with all extracted messages.
  • ICU Expressions: Handle pluralization (`{msgCount, plural, =0 {No msgs} one {1 msg} other {# msgs}}`) and alternate selections (`{gender, select, male {He} female {She} other {They}}`).
Think of AOT i18n like publishing a translated book. You print a completely different physical book for English and French. The reader doesn't waste time translating while reading; they just pick up the French book.
Runtime Translation (ngx-translate / Transloco)

If you cannot use compile-time translation (e.g., you need users to switch languages instantly without a page reload, or translations come from a database API), you must use a runtime library like Transloco or ngx-translate.

  • Loads translations asynchronously (JSON files).
  • Uses Pipes to display data: `{{ 'HOME.TITLE' | transloco }}`.
  • Slightly worse performance than AOT because Angular has to watch bindings and evaluate them at runtime.
