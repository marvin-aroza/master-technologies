App Router Deep Dive

The modern standard for Next.js routing, introducing Server Components, Streaming, and layout persistence.

File Conventions
  • page.tsx: The unique UI of a route. Makes the route publicly accessible.
  • layout.tsx: Shared UI for a segment and its children. Persists across route changes without re-rendering.
  • loading.tsx: Optional suspense fallback UI. Next.js automatically wraps `page.tsx` with `}>`.
  • error.tsx: Error boundary. Must be a Client Component (`"use client"`). Catches unexpected errors in children.
  • not-found.tsx: Displayed when a `notFound()` function is thrown or a URL doesn't match any route.
  • route.ts: A Server-side API endpoint (Route Handler).
  • default.tsx: Used as a fallback for unmatched parallel route slots (like modals or complex dashboards).
Advanced Routing Patterns
  • Parallel Routes (`@folder`): Allows simultaneously or conditionally rendering one or more pages within the same layout (e.g., a dashboard with an `@analytics` slot and a `@team` slot).
  • Intercepting Routes (`(.)folder`): Intercepts navigation to a different route and renders its content in the current layout context (e.g., clicking a photo in a feed opens a Modal instead of navigating to the photo's dedicated page URL. If refreshed, it loads the dedicated page).
  • Route Groups (`(folder)`): Folders in parentheses are ignored in the URL path. They are used purely to organize files logically or to opt specific routes into a shared `layout.tsx`.

Middleware & Authentication

Securing routes, redirecting traffic, and handling user sessions at the Edge.

Next.js Middleware

Middleware allows you to run code before a request is completed. Based on the incoming request, you can modify the response by rewriting, redirecting, modifying the request or response headers, or responding directly.

  • Edge Runtime: Middleware runs on the Vercel Edge Network by default. It must be ultra-fast and cannot use standard Node.js APIs (like `fs` or `mongoose`).
  • Matcher: Using `export const config = { matcher: ['/dashboard/:path*'] }` ensures middleware only runs on specific routes, saving costs and execution time.
  • Common Use Cases: Authentication checks, Geolocation (showing specific country pricing), A/B Testing, Bot protection, and rate limiting.
Authentication Patterns

Authentication in Next.js App Router usually relies on reading cookies on the server before rendering.

  • NextAuth.js (Auth.js) v5: The de facto standard library. Supports OAuth (Google, GitHub), Email/Passwordless, and Credentials. Integrates perfectly with Middleware to protect routes.
  • Server-Side Protection: Wrap protected pages in a check like `const session = await auth(); if (!session) redirect('/login');`.
  • Client-Side Protection: Less secure. Relies on `useSession()` hook. The user might see a flash of the protected page before being redirected. Always prefer Server-Side.

Deployment & Ecosystem

Taking Next.js to production efficiently.

Vercel & Output Types
  • Next.js on Vercel: Zero-config deployment. Automatically splits Server Components into Edge Functions, caches static pages on the global CDN, and handles image optimization perfectly.
  • output: 'standalone': Set in `next.config.js`. This traces `import` calls and outputs ONLY the necessary files needed to run the app in production. This is absolutely critical for deploying Next.js efficiently via Docker containers on AWS/GCP, reducing image sizes drastically.
  • output: 'export': Generates a strictly static site (like plain HTML/CSS). Server Components run at build time. Dynamic server logic (like `cookies()` or `headers()`) will throw errors. Used when deploying to S3, GitHub Pages, or Netlify static hosting.
