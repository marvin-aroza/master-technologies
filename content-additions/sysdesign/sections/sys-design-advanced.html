Scalable Architecture Fundamentals

The core principles of designing systems that handle massive traffic and data without failing.

Vertical vs Horizontal Scaling
  • Vertical Scaling (Scale Up): Adding more power (CPU, RAM) to a single existing machine.
    Pros: Extremely simple, no distributed system complexity.
    Cons: Hard hardware limits, single point of failure (SPOF), requires downtime to upgrade.
  • Horizontal Scaling (Scale Out): Adding more machines into your pool of resources.
    Pros: Infinite scalability, fault tolerance, no hardware limits.
    Cons: Data inconsistency risks, complex routing (Load Balancers needed), difficult state management.
Load Balancing

A load balancer sits between clients and servers, distributing incoming traffic across multiple healthy servers.

  • Algorithms: Round Robin (sequential), Least Connections (send to server with fewest active connections), IP Hash (ensures a specific user always hits the same server).
  • Layer 4 vs Layer 7: L4 balances based strictly on IP and TCP ports (fast, dumb). L7 balances based on HTTP headers, URLs, and cookies (slower, smart).
  • Health Checks: The LB actively pings servers. If a server dies, the LB stops sending traffic to it automatically.
CAP Theorem

In a distributed data store, you can only mathematically guarantee exactly two of the following three traits:

  • Consistency: Every read receives the most recent write or an error.
  • Availability: Every request receives a (non-error) response, without the guarantee that it contains the most recent write.
  • Partition Tolerance: The system continues to operate despite an arbitrary number of messages being dropped/delayed by the network between nodes.

Note: Because network partitions (P) are a physical reality of the internet and cannot be avoided, real-world systems must choose between AP (Availability) or CP (Consistency).

Data Storage & Scaling

Managing state when data becomes too large for a single machine.

Relational (SQL) vs Non-Relational (NoSQL)
  • SQL (PostgreSQL, MySQL): Structured schemas, ACID compliance, complex joins. Perfect for highly structured, predictable data where consistency is critical (e.g., Financial transactions). Hard to horizontally scale.
  • NoSQL (MongoDB, DynamoDB, Cassandra): Flexible schemas, eventual consistency, no complex joins. Designed specifically to be horizontally partitioned from day one. Perfect for massive volume, unstructured data, or rapid prototyping (e.g., Social media feeds, IoT sensor logs).
Database Scaling Techniques
  • Read Replicas: One "Master" database handles all writes. Multiple "Slave" databases copy the master and handle all reads. Fixes read-heavy bottlenecks.
  • Sharding (Data Partitioning): Splitting a single large database across multiple smaller databases based on a Shard Key (e.g., User ID). If `ID < 1000` goes to DB_A, `ID > 1000` goes to DB_B. Extremely fast, but makes cross-shard joins nearly impossible.
  • Indexing: Creating a specific data structure (like a B-Tree) on a database column to allow O(log n) lookups instead of O(n) full table scans. Speeds up reads massively, but slows down writes (because the index must be updated on every write).
Caching Strategies

Storing expensive computations or database queries in ultra-fast, in-memory stores like Redis or Memcached.

  • Cache-Aside: App queries Cache. If miss, app queries DB, then writes to Cache, then returns to user. Best generic strategy.
  • Write-Through: App writes immediately to Cache AND DB at the same time. Cache is always perfectly consistent with DB. Slower writes.
  • Write-Back: App writes ONLY to Cache and returns success immediately. Cache asynchronously syncs to the DB later. Insanely fast writes, but high risk of absolute data loss if the cache node crashes before syncing.
