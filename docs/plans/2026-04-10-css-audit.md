## Cross-cutting findings

- The current CSS topic has 12 tabs and 12 sections, with 102 Q&A items total when the 11 `css-box` questions are counted alongside the 91-item `css-iq` bank.
- The topic has strong raw material, but it is still a mixed curriculum rather than a clean mastery path. Core layout, modern platform features, architecture, accessibility, performance, and interview prep are all present in the same file.
- The current shape is most useful as a source bank, not as a teaching sequence. The rewrite should preserve the good material and move it into a clearer progression instead of deleting it.
- The biggest structural problem is overlap. `css-modern`, `css-2025`, `hc-modern`, and `css-tricks` all cover modern CSS features from slightly different angles, while `css-arch`, `hc-a11y`, and `web-perf` push into later applied concerns, and `css-cs` plus `css-iq` duplicate recap and interview material.
- The CSS topic already contains several 2026-relevant ideas, but they are scattered across tabs and not framed with the right browser-model ordering yet.
- The `css-iq` tab label still says `Interview Q&A (91)`, but the topic overall has 102 Q&A items because `css-box` also includes interview-style questions. That is a maintenance signal, not just a display detail.

## What The Current Topic Gets Right

- `css-box` is the strongest core foundation because it already covers the box model, flexbox, grid, positioning, transforms, and several practical interview checks.
- `css-selectors` is a clean base for selectors, combinators, pseudo-classes, and pseudo-elements.
- `css-modern` contains the most useful modern CSS fundamentals: custom properties, container queries, selectors, animations, and responsive design.
- `css-2025` has the right future-facing platform features, especially anchor positioning, view transitions, cascade layers, and relative color syntax.
- `css-tricks` has useful applied techniques that should survive, but the chapter boundary is too broad for the final structure.
- `css-arch` already points in the right direction for architecture, specificity, and performance, even though it mixes core and applied concerns.
- `css-cs` is valuable as fast recall material, but it is too mixed to remain a primary teaching chapter.
- `css-iq` is large enough to support distributed interview reinforcement once it is broken apart and deduplicated.

## What Is Mixed Or Duplicated

- `css-modern` and `css-2025` overlap heavily on modern platform features. The rewrite should collapse them into one coherent modern-CSS progression instead of treating them as separate feature dumps.
- `hc-modern` repeats a subset of modern selector and responsiveness material that already belongs in `css-modern` and `css-selectors`.
- `css-tricks` mixes real production techniques with feature showcase content. Its useful ideas should be redistributed into the relevant core or later chapters.
- `css-arch` mixes architecture, methodology, and performance in one section. The architecture and methodology part belongs much later than the core layout chapters.
- `hc-a11y` belongs in the CSS rewrite only as accessibility-safe styling guidance. Any broad HTML-semantics framing should be pared back or moved out.
- `web-perf` is useful, but it is an applied performance appendix rather than a core CSS teaching chapter.
- `css-cs` duplicates material that should be split across chapter recaps, reference notes, and interview drills.
- `css-props` is a reference appendix, not a mastery chapter. Its content should be mined for recap support and quick lookup, not treated as a main path.
- `css-iq` centralizes interview prep too much. The strongest questions should be distributed into the chapters, with only the hardest review material left for the final capstone.

## 2026-Sensitive Corrections

- `@layer` should be taught as part of cascade ordering, not as a styling namespace or component boundary. It sits in the cascade before specificity, and `revert-layer` should be introduced with it.
- `@scope` should be framed as scoped selector matching between a root and an optional limit, not as Shadow DOM replacement or component encapsulation.
- `:has()` should be taught as a relational selector, not casually as a "parent selector". It is powerful for state-aware styling, but the rewrite should also note selector complexity and performance awareness.
- Container queries should be separated from container units. `@container` queries component size, while `cqw`, `cqh`, `cqmin`, and `cqmax` are size-relative units that depend on the query container.
- Scroll-driven animations should be described as native timeline-driven animation, not as a generic replacement for every JS animation use case.
- View transitions should be framed as UI-state transition tooling for same-document and cross-document transitions, not as a universal animation framework.
- Anchor positioning should be presented as native tethering for popovers, tooltips, and floating UI, not as a full replacement for all absolute positioning.
- Rendering-cost guidance needs tighter browser-model language: layout changes geometry, paint fills pixels, and composite combines layers. Animating `transform` and `opacity` is usually cheapest, but `will-change` is only a hint and should be used sparingly.

### Keep In Core CSS

- `css-box`
  - box model
  - sizing
  - overflow
  - positioning fundamentals
  - flexbox basics
  - grid basics
- `css-selectors`
  - selectors
  - combinators
  - pseudo-classes
  - pseudo-elements
  - selector reasoning and specificity setup
- The core parts of `css-modern`
  - CSS custom properties
  - modern selectors
  - container query fundamentals
  - animations and transitions as browser primitives
  - responsive design toolkit ideas

### Move To Later CSS Chapters

- `css-modern`
  - deeper modern selector behavior
  - more advanced responsive patterns
  - feature usage that depends on applied component design
- `css-tricks`
  - CSS functions
  - scroll snap
  - shapes and clip paths
  - system integration details such as `accent-color` and `color-scheme`
- `hc-a11y`
  - accessibility-safe styling
  - focus visibility
  - semantic boundaries that matter for CSS-driven UI
- `web-perf`
  - INP, LCP, and CLS framing
  - asset prioritization
  - image and font cost
  - browser work that belongs in the rendering-cost chapter
- The browser-model and performance parts of `css-arch`
  - specificity as a system
  - performance thinking
  - large-scale CSS architecture as a later applied chapter

### Reserve For Final Ecosystem Tradeoff Material

- `css-arch`
  - CSS architecture methodology comparisons
  - BEM
  - utility-first discussion
  - CSS Modules
  - CSS-in-JS
  - design-system scaling tradeoffs
- Any methodology comparison that is really about ecosystem choice rather than CSS itself
  - utility-first vs classic CSS
  - component-scoped CSS vs global CSS
  - runtime styling vs build-time styling

### Merge Into Recap, Reference, Or Interview Sections

- `css-cs`
  - keep the strongest fast-recall points
  - break the content into chapter recaps instead of one mixed cheat sheet
  - preserve concise browser-model reminders
- `css-props`
  - keep as a reference appendix
  - use it as lookup support for chapter recaps and drills
- `css-iq`
  - distribute the best questions across the chapters
  - keep the hardest questions for the final capstone
  - preserve the CSS-only interview material that the HTML rewrite intentionally left behind, especially the items about `box-sizing`, `z-index`, Flexbox alignment, BEM, stacking contexts, container units, and grid-only mechanics

  - move them into the advanced platform CSS chapter rather than leaving them as a standalone feature tab

  - preserve only the highest-signal recall prompts
- `css-iq`

  - leave the final chapter as a capstone, not a dump

## What Must Be Preserved

- The strongest layout foundations from `css-box`, especially the box model, flexbox, grid, and positioning examples.
- The selector and specificity material from `css-selectors` and `css-modern`.
- The modern CSS features that are now production-ready enough to teach as part of the platform, including custom properties, container queries, `@layer`, `@scope`, and `:has()`.
- The applied browser-performance material around layout, paint, composite, containment, and `will-change`.
- The accessibility-safe styling material, but rewritten so it stays CSS-first instead of turning into a generic accessibility chapter.
- The CSS-only interview material preserved during the HTML rewrite, especially the questions about `box-sizing`, `z-index`, Flexbox alignment, BEM, stacking contexts, container units, and grid-only mechanics.
- The strongest fast-recall notes from `css-cs`, but only after they are split into chapter recaps and the final mastery chapter.

## Summary

The current CSS topic already contains enough material for a strong 2026 mastery track. What it does not yet have is clean chapter ownership. The rewrite should keep the core CSS foundations, move the applied platform and performance material into later chapters, reserve methodology tradeoffs for the final ecosystem section, and redistribute the interview bank so learning is reinforced throughout the topic instead of being trapped in one endcap.
